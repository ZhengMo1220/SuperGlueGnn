一、系統概覽

SuperGlue 是一個特徵點匹配系統：給定兩張影像，找出彼此之間在空間上對應的像素位置（特徵點對）。整個系統是兩個神經網路的串接：


① SuperPoint（前端）
CNN，在每張圖上找出最顯著的角點（關鍵點），並計算每個點的 256 維特徵向量（描述子）。類似「每個點的指紋」。
② SuperGlue（中端）
圖神經網路（GNN），接收兩張圖的所有描述子，透過 attention 機制讓特徵互相參考，最後用最優傳輸算法找出最佳匹配。

應用場景：相機定位、三維重建（SfM）、影像配準、視覺里程計。你的場景：找出前方相機（cf）與側邊相機（cs）拍攝的標定棒在兩個視角之間的對應點。

二、推理 Pipeline（資料流向）
📷 影像對
cf/1.jpg + cs/1.jpg
→
🔍 SuperPoint
跑兩次（每張各一次）
→
📍 關鍵點 + 描述子
N×2 + N×256
→
🔗 SuperGlue
GNN 18 層 Attention
→
✅ 匹配結果
matches + confidence

步驟模型輸入輸出關鍵超參數
1. 影像前處理 utils.read_image() 彩色/灰階 JPG 灰階 float32 tensor --resize
2. 關鍵點偵測 SuperPoint 灰階影像 [1×1×H×W] keypoints [N×2], scores [N], descriptors [256×N] --max_keypoints, --keypoint_threshold, --nms_radius
3. GNN 匹配 SuperGlue 兩張圖的 keypoints + descriptors matches0 [N], matching_scores0 [N] --sinkhorn_iterations, --match_threshold
4. 結果儲存 — matches, conf *_matches.npz —
5. 視覺化（選用） make_matching_plot() 兩張圖 + 匹配點對 *_matches.png --viz, --fast_viz

⚙️ 所有參數說明
參數預設值類型說明 / 你的場景建議
── 路徑設定 ──
--input_pairs assets/scannet_sample_pairs_with_gt.txt 必填 配對清單路徑。你的場景：Bullpen_CalibrationBar/pairs.txt
--input_dir assets/scannet_sample_images/ 必填 影像根目錄。你的場景：Bullpen_CalibrationBar（pairs.txt 裡的路徑相對於此）
--output_dir dump_match_pairs/ 選填 輸出目錄。建議：Bullpen_CalibrationBar/output
--max_length -1（全部） 選填 最多處理幾對。測試時設 2，確認沒問題再跑全部
── 影像處理 ──
--resize 640 480 選填 縮放模式三選一：
640 480 → 強制指定寬×高
1280 → 最長邊縮到 1280
-1 → 不縮放（原圖）
建議起步：640 480，匹配少時試 1280
--resize_float 不啟用 旗標 縮放時用浮點數精度。戶外場景官方建議加此旗標
── 模型選擇（SuperPoint） ──
--max_keypoints 1024 選填 每張圖最多偵測幾個關鍵點。-1 = 不限。匹配少時試 2048 或 4096
--keypoint_threshold 0.005 選填 關鍵點偵測閾值。越低 = 偵測越多點（但也更多雜訊）
--nms_radius 4 選填 非極大值抑制半徑（像素）。戶外建議改 3
── 模型選擇（SuperGlue） ──
--superglue indoor ⚠️ 重要 indoor：室內場景（ScanNet 訓練）
outdoor：戶外場景（MegaDepth 訓練）
你的 Bullpen 是戶外 → 用 outdoor
--sinkhorn_iterations 20 選填 Sinkhorn 最優傳輸迭代次數。越高越精確但越慢，20 已夠用
--match_threshold 0.2 選填 越低 = 保留更多（但較不確定）的匹配。範圍 0.1～0.5
── 輸出控制 ──
--viz 不啟用 旗標 輸出視覺化圖片（.png）。建議每次都加
--fast_viz 不啟用 旗標 用 OpenCV 繪圖（比 Matplotlib 快很多）。需搭配 --viz。建議加
--show_keypoints 不啟用 旗標 在圖上顯示所有偵測到的關鍵點（含未匹配的）。除錯用
--eval 不啟用 旗標 啟用定量評估（需要 GT：K 內參 + T 姿態）。目前先不用
--cache 不啟用 旗標 跳過已有 .npz 的配對。重跑時節省時間
--force_cpu 不啟用 旗標 強制用 CPU。預設自動偵測並使用 GPU（你有 RTX 5080，不需加）
⚠️
旗標組合限制：--fast_viz 必須搭配 --viz；--opencv_display 必須同時搭配 --viz 和 --fast_viz；--fast_viz 不支援 --viz_extension pdf。
五、models/matching.py — Pipeline 整合器
🔗
Matching 類別
models/matching.py → class Matching(torch.nn.Module)

把 SuperPoint 和 SuperGlue 串在一起，對外只需要傳入影像，內部自動完成所有步驟。

from models.matching import Matching

config = {
    'superpoint': {
        'nms_radius': 4,
        'keypoint_threshold': 0.005,
        'max_keypoints': 1024
    },
    'superglue': {
        'weights': 'outdoor',       # 你的場景用 outdoor
        'sinkhorn_iterations': 20,
        'match_threshold': 0.2
    }

matching = Matching(config).eval().to('cuda')

輸入 data 字典

image0Tensor [1×1×H×W]
image1Tensor [1×1×H×W]
keypoints0 （可選）若提供則跳過 SuperPoint

輸出 pred 字典

七、models/superglue.py — GNN 匹配網路
🔗
SuperGlue
models/superglue.py → class SuperGlue(nn.Module)
內部運作流程
1
Keypoint Encoding
將關鍵點的座標（x,y）和偵測分數（score）透過 MLP 融合進 256 維描述子。讓描述子同時包含「視覺外觀」和「位置」資訊。
2
Attentional GNN（18 層）
交替執行 self-attention（同張圖內的點互相參考）和 cross-attention（跨兩張圖的點互相參考）。讓每個點的特徵能「看到」整張圖的上下文。共 9 次 self + 9 次 cross。
3
Optimal Transport（Sinkhorn 算法）
計算兩組特徵的 N×M 相似度矩陣，用 Sinkhorn 迭代找到一個「最優傳輸」分配，讓每個點最多對應到一個點（mutual exclusivity）。
4
閾值過濾
信心分數 > match_threshold 且互相最佳匹配才保留。最終輸出 matches0[i] = j 表示圖0的第 i 個點對應到圖1的第 j 個點（-1 = 無匹配）。
重要參數
參數預設值說明
weights 'indoor' 選 'indoor'（ScanNet）或 'outdoor'（MegaDepth）。你的場景用 outdoor
match_threshold 0.2 信心分數低於此值的匹配會被丟棄。降低 = 保留更多但品質可能較差的匹配
sinkhorn_iterations 100（訓練）/ 20（推理） 迭代次數。20 對推理已足夠，不需更改
GNN_layers ['self','cross'] × 9 18 層交替 attention。固定值，不可更改（由訓練決定）

九、pairs.txt 格式說明
模式一：純匹配（2 欄）— 你目前使用的
格式
name0 name1
你的 Bullpen_CalibrationBar/pairs.txt
cf/1.jpg cs/1.jpg
cf/2.jpg cs/2.jpg
...（共 21 行）
💡
路徑相對於 --input_dir。你設 --input_dir Bullpen_CalibrationBar，所以寫 cf/1.jpg 就會讀到 Bullpen_CalibrationBar/cf/1.jpg。
模式二：含 GT 評估（38 欄）— 未來需要
格式（每行 38 個以空格分隔的值）
name0 name1 rot0 rot1  K0_11 K0_12 K0_13 K0_21 K0_22 K0_23 K0_31 K0_32 K0_33  K1_11 ... K1_33  T_11 T_12 T_13 T_14 T_21 ... T_44
欄位數量說明
name0, name12影像相對路徑
rot0, rot12EXIF 旋轉（0/1/2/3）。無資料填 0
K09Camera 0 的 3×3 內參矩陣，逐行展平
K19Camera 1 的 3×3 內參矩陣，逐行展平
T_0to1164×4 相對姿態（cam1 相對 cam0），逐行展平
K 矩陣展平方式
K = [[fx,  0, cx],
     [ 0, fy, cy],
     [ 0,  0,  1]]

# 展平 → 9 個數值：
fx 0 cx 0 fy cy 0 0 1
T_0to1 展平方式
T_0to1 = [[R11 R12 R13 tx],
           [R21 R22 R23 ty],
           [R31 R32 R33 tz],
           [  0   0   0  1]]

# 展平 → 16 個數值（逐行）：
R11 R12 R13 tx R21 R22 R23 ty R31 R32 R33 tz 0 0 0 1
十、.npz 輸出格式說明
*_matches.npz（每次匹配都會產生）

keypoints0

形狀[N, 2]
單位像素座標 (x, y)
對應cf 影像的關鍵點

keypoints1

形狀[M, 2]
單位像素座標 (x, y)
對應cs 影像的關鍵點

形狀[N]
意義kp0[i] 對應到 kp1[matches[i]]

Python 讀取範例
import numpy as np

npz = np.load('Bullpen_CalibrationBar/output/1_1_matches.npz')

kpts0 = npz['keypoints0']       # [N, 2]  cf 影像的所有關鍵點 (x, y)
kpts1 = npz['keypoints1']       # [M, 2]  cs 影像的所有關鍵點 (x, y)
matches = npz['matches']         # [N]     matches[i] = j 或 -1
conf = npz['match_confidence']  # [N]     0~1 信心分數

# 篩選有效匹配
valid = matches > -1
matched_kpts0 = kpts0[valid]                  # cf 的匹配點
matched_kpts1 = kpts1[matches[valid]]         # cs 的對應點
matched_conf  = conf[valid]                   # 對應信心分數

print(f'總關鍵點：{len(kpts0)} (cf), {len(kpts1)} (cs)')
print(f'有效匹配數：{valid.sum()}')
*_evaluation.npz（加 --eval 才會產生）
欄位型態說明
error_Rscalar（度）旋轉誤差（與 GT 比較）
error_tscalar（度）平移方向誤差（與 GT 比較）
precision0~1對極誤差 < 5×10⁻⁴ 的匹配比例
matching_score0~1正確匹配數 / kp0 總數
num_correctint對極誤差 < 5×10⁻⁴ 的匹配數
epipolar_errors[M]每個匹配的對極幾何誤差

十四、後續步驟路線圖
1
✅ 現在：跑匹配視覺化，觀察結果
執行指令③（或②），打開 output/ 裡的 PNG，確認匹配點數量和分布是否合理。如果點太少，嘗試調整 --max_keypoints 或 --match_threshold。
2
🔧 下一步：取得相機內參 K
用棋盤格拍多角度照片（≥10 張），用 OpenCV calibrateCamera() 取得 K 和 dist。這是做 --eval 評估的前提。reprojection error < 1px 才算可信。
3
🔧 取得相對姿態 T_0to1
對每對影像，用已知 3D 座標的標記點 + solvePnP() 分別取得 cf 和 cs 相機的外參，再計算相對姿態 T_0to1 = T_cam1 @ inv(T_cam0)。
4
🔧 生成 38 欄 pairs_with_gt.txt
將 K 和 T_0to1 寫入 pairs.txt，格式為 38 欄，就可以啟用 --eval 模式，取得 AUC、Precision、Matching Score 等定量指標。

❓ 問題 Q&A（實驗過程遇到的問題整理）

更新於 2026-05-24 · 按時間順序整理，包含問題原因與解決狀態

已解決 Q1｜Windows 終端機顯示亂碼 / UnicodeEncodeError

問題：在 Windows cp950 環境下，Python print 含 emoji（✅⚠️❌）或中文時出現 UnicodeEncodeError 或亂碼。

原因：Windows 預設終端機編碼為 cp950，不支援 Unicode emoji；中文字元也可能因編碼不匹配而亂碼。

解法：將所有 print 改為純 ASCII 標記（[OK]、[WARNING]、[FAIL]），並將所有 print 語句改為英文輸出。

已釐清 Q2｜GT F matrix 的 Sampson distance 異常高（~24000）

問題：用 GT F matrix 對 SuperGlue 的 inlier 點計算 Sampson distance，結果高達 24038，但估計的 F 只有 0.56，差距極大。

原因：GT F 是在原圖解析度（1920×1084）下計算的，而第一次 SuperGlue 輸入是 resize 後的 640×480。兩組點的座標尺度不同，GT F 套用在小解析度的點上自然誤差極大。這不是 pipeline 錯誤。

解法：改用 1920×1084 原圖解析度跑 SuperGlue，座標系一致後 Frobenius norm 從 0.135 降到 0.021，GT Sampson 也降低。

已釐清 Q3｜SuperGlue 偵測不到保麗龍球

問題：場景中用來當標定點的白色保麗龍球，SuperGlue 幾乎完全無法匹配。

原因：SuperPoint（SuperGlue 的前端）偵測角點與紋理邊緣。白色保麗龍球表面無紋理、無角點，SuperPoint 不會在球面放 keypoint。這是 SuperPoint 的設計限制，不是 bug。

影響：SuperGlue 的 inlier 全部來自背景紋理（草皮、圍網、棒子邊緣），這些點在幾何上可以建立正確的 F matrix，但無法提供精確的標定點座標。保麗龍球的用途應改為 PnP（已知 3D 座標），而非 SuperGlue 的 keypoint 來源。

已釐清 Q4｜不同解析度算出的 F matrix 數值差很多，是錯的嗎？

問題：640×480 和 1920×1084 跑出的 F matrix 數值完全不同，讓人擔心結果錯誤。

原因：F matrix 的數值會隨座標尺度縮放，不同解析度的 F 直接比數字沒有意義。正確的比較方式是 Frobenius norm（normalize 後的矩陣差距）。

結論：1920×1084 的 F 與 GT F 的 Frobenius norm 為 0.021（極佳，< 0.1 為接近）。幾何上正確，只是數值尺度不同。

已確認 Q8｜最終目標：3D 骨架節點估計 → 投球動作分析

目標：透過雙相機（側面 cs + 正面 cf）的 2D 骨架點（ViT-Pose 輸出），經三角測量重建投手的 3D 骨架節點座標，進而計算物理特徵（腳踝距離、肩外旋角度、手腕速度等），切割投球動作 4 個階段。

待確認 Q5｜GT F matrix 只有 22/120 inliers，是否可靠？

問題：人工標記的 120 個點，RANSAC 後只剩 22 個 inlier（18.3%），GT F 是否可信？

原因：人工標記的 6 個點/張分布在保麗龍球位置，這些點的 epipolar 一致性本來就有限（標定點不是為 F matrix 設計的，分布也不夠分散）。22 個 inlier 算 F 勉強夠用但不穩定。

結論：SuperGlue 的 F（290 inliers）比 GT F（22 inliers）更可靠，inlier 數量多 13 倍。GT F 僅供對照參考。

核心瓶頸 Q6｜Pose Estimation 的 R 算出 160°，但實際只有 ~90°

問題：從 F → E → recoverPose 得到的旋轉角度約 160°，但兩台相機實際夾角約 90°。

原因（兩個）：

  1. 兩台相機 FoV 差距 5 倍：cf（50mm 長焦，FoV 13.8°）vs cs（8mm 廣角，FoV 68.7°）。SuperGlue 設計上假設兩張圖有大量視野重疊，FoV 差 5 倍時匹配前提失立，inlier 雖多但幾何約束不正確。
  2. cf 主點偏移：cx=869（影像中心應為 960），偏移 91px，代表標定時影像可能有裁切或感光器對齊問題，影響 E = K2ᵀFK1 的計算。

驗證：cs 的 inlier 全部集中在影像左側 25%（X: 8~475px），對應 cf FoV 投影範圍，確認是 FoV 不匹配造成的系統性誤差。

核心瓶頸 Q7｜SuperGlue 在不同焦距相機間的根本限制

問題：SuperGlue 能否正確處理焦距差距 5 倍的兩台相機？

分析：SuperGlue（SuperPoint + GNN matching）在設計上針對相近視野、相似解析度的影像對。當一台看到的場景是另一台的 5 倍大時：

  • 同一個場景物件在兩張圖中的尺寸差 5 倍 → descriptor 不相似
  • cs 影像大部分區域在 cf 中根本看不到 → 大量誤匹配
  • F matrix 雖然數值上接近（Frobenius norm 0.021），但用來 recoverPose 仍然失敗

結論：這是此相機配置下 SuperGlue 的根本限制，非參數調整可以解決。

核心瓶頸 Q9｜為什麼 SuperGlue 的 inlier 比人工多，但 F matrix 差異更大？

問題：SuperGlue 找到 290 個 inlier（遠多於人工的 22 個），但算出的 Pose（R, t）卻是錯的，而人工標記的 F 雖然 inlier 少，幾何上卻更接近真實。這是為什麼？

關鍵原因：inlier 多 ≠ 正確，要看點的「語義品質」和「分布」

SuperGlue（背景紋理） 人工標記（保麗龍球）
inlier 數量 290（多） 22（少）
點的來源 草皮、圍網、棒子邊緣（背景） 保麗龍球球心（語義明確）
FoV 問題 cs 的 inlier 集中在左側 25%（FoV 不匹配造成系統性偏差） 點位已考慮兩台相機的共同可見區域
F matrix 品質 Frobenius norm 0.021（幾何數值接近） GT F（基準）
Pose 結果 R ≈ 160°（錯誤，應為 ~90°） 未直接驗證，但作為 GT 參考

核心矛盾說明：

  • F matrix 幾何上正確（0.021）但 Pose 錯誤——因為 F matrix 描述的是「兩張圖中點的對極幾何關係」，它可以在 FoV 嚴重不匹配的情況下數值上收斂，但這個 F 對應的幾何並非真實的相機位置關係。
  • inlier 多反而是壞事——290 個點都集中在 cs 影像的左側 25%，它們一致性很高（所以 RANSAC 選中了），但這個「一致性」是 FoV 不匹配造成的系統性偏差，不是真實的幾何約束。
  • 類比：好比你用一把歪掉的尺量了 290 次，每次結果都一致（inlier 多），但量出來的值就是錯的。人工標記雖然只有 22 次，但用的是正確的尺。
核心瓶頸 Q10｜SuperGlue 在此場景的根本限制總結

以下是 SuperGlue 在本專案中所有已確認的限制，確認這些限制真的無法克服後，才考慮換新方法：

根本診斷（更新 2026-05-23） Q14｜問題出在哪個環節？特徵點、匹配、還是 F→E 轉換？

問題：3D 骨架重建失敗，問題是出在 SuperPoint 特徵點偵測、SuperGlue 匹配，還是後處理的 F→E→R,t 計算？

SuperPoint 找特徵點     → 正常（背景紋理角點正確偵測）
SuperGlue 找 corresponding → ❌ 此步驟是根本問題（F 品質不足，見下方分析）
RANSAC 算 F            → ❌ Frobenius 0.021 看似接近，但 E 奇異值比嚴重偏離 1:1
F → E = K_csT F K_cf   → K 轉換本身沒有問題（見實驗驗證）
E → recoverPose → R,t   → ❌ SG：R ≈ 160°；人工點：R ≈ 84°（正確）
triangulatePoints       → ❌ 3D 骨架躺平（因 SG 的 R 錯誤）

🔬 實驗驗證（2026-05-23 新增）：對人工標記 120 個點做完整 F→E→R,t pipeline 驗證：

人工點 E 奇異值比：sv[0]/sv[1] = 1.047（幾乎理想的 1:1）
人工點 Pose：Roll=65.8°, Pitch=-84.0°（正確！cs 在 cf 右側 ~90°）, Yaw=-66.0°
SG 點 Pose：Roll=160°, Pitch=-28°（錯誤，偏差 >60°）

修正後的根本原因：K_cf 條件數不是問題，問題是 SuperGlue 的 F matrix 品質不夠好。

  • 對應點是對的：視覺化可見連線多在圍網同一位置，SuperGlue 確實找到了正確的幾何對應
  • 但 F matrix 退化：大量特徵點集中在圍網（近似平面），平面退化（planar degeneracy）讓 RANSAC 算出的 F 幾何上不精確
  • E 奇異值比嚴重偏離：SG 的 F 算出的 E 奇異值比例遠大於 1.0，recoverPose 強制矯正引入大角度誤差
  • 人工點因為分布在場景多個位置（棒子、投手板、牆面），不落在單一平面，F 品質更好

結論：這是 SuperGlue matching 的問題，換用對平面退化更穩健的匹配方式有機會改善。

  • 方案 1（最直接）：增加人工標記點 — 從 6 點/對增加到 15-20 點，確保分布在不同景深，直接算出好的 F
  • 方案 2：LoFTR / LightGlue — Dense matching 對低紋理區域及 FoV 差異更穩健，有機會產生更好分布的 corresponding points
  • 方案 3（不可行）：ChArUco board — 兩機夾角 ~90°，無法同時拍到同一塊板子，已排除
討論中 Q15｜如何繼續推進？下一步選擇

根本問題已確認（見 Q14）：SuperGlue 的 F matrix 因特徵點集中在平面（圍網）而退化。以下是三個可行方向（ChArUco 已確認不可行，因兩機夾角 ~90°）：

方向 A｜增加人工標記點

目前每對只有 6 個點，增加到 15–20 點，確保分布在不同景深（棒子、投手板、地面、牆面），可直接改善 F matrix 品質，成本最低。

方向 B｜LoFTR 論文

Dense matching（CVPR 2021），不需要 keypoint，對 FoV 差異（5×）更有容忍度，有機會找到分布在不同深度的 corresponding points，有開源實作。

方向 C｜LightGlue 論文

SuperGlue 原作者的改進版（ICCV 2023），matching 更快更準，配合 DISK / ALIKED 特徵點偵測器，對低紋理區域有改善。

進行中 Q16｜LoFTR 測試結果：能否解決 SuperGlue 的 matching 問題？

測試方法：用 kornia 的 LoFTR（outdoor 權重）對 pair 08 做 dense matching，和 SuperGlue 做 Pose 比較。LoFTR 不需要 SuperPoint，直接在整張圖做 attention 輸出 corresponding points。

已測試・失敗 Q17｜LightGlue 測試結果：keypoint-based 在 FoV 差距 5× 下完全失效

背景：LightGlue（ICCV 2023）是 SuperGlue 原作者（ETH Zurich）的新一代 keypoint matching 論文，搭配 SuperPoint / DISK / ALIKED 特徵點偵測器，速度和精度均優於 SuperGlue。

SuperPoint + LightGlue  → matches=7， sc>0.5=0，Pitch=11.2° ❌
DISK + LightGlue       → matches=20，sc>0.5=0，Pitch=-28.6° ❌
ALIKED + LightGlue     → matches=38，sc>0.5=2，Pitch=3.6° ❌
LoFTR outdoor（對照）  → matches=73，sc>0.5=15，Pitch=-82.0° ✅

根本原因：LightGlue 和 SuperGlue 都是 keypoint-based，先偵測角點再配對。當 FoV 差距 5× 時，同一個場景點在兩張圖中的大小差 5 倍，descriptor 完全不相似，高信心的 match 幾乎找不到，換再好的 matching 網路也無法解決。

結論：此場景下 detector-based 方法（SuperGlue / LightGlue）均無效，必須使用 detector-free 方法（LoFTR）才能在 FoV 差距大時找到有意義的 corresponding points。
